Проект на семестр
Часть 1 «Проектирование доменно-ориентированной архитектуры и инфраструктуры как код».

  Задача: Вместо установки локальных инструментов — развернуть основу платформы в облаке (Yandex.Cloud / AWS / GCP) с помощью Terraform.

  Действия:

1.    Определить домены данных в рамках кейса (напр., academic_performance, campus_infrastructure, student_engagement).

2.    Написать Terraform-конфигурации для создания: Object Storage (S3-совместимого), управляемого Kafka-кластера, сервера для ВМ.

3.    Развернуть Airflow или Prefect в Docker на созданной ВМ.

4.    Создать схему Data Product для одного домена по принципам Data Mesh (описать интерфейсы, SLA, метрики качества).



Часть 2 «Разработка отказоустойчивых ETL/ELT пайплайнов с Observability».

   Задача: Реализовать загрузку данных из источников в Raw-слой (Data Lake) с контролем качества.

   Действия:

1.    Написать DAG в Airflow/Prefect, который загружает данные из разных источников (например, симуляция API LMS + CSV-выгрузки) в S3 в формате Parquet.

2.    Внедрить этапы валидации данных с помощью Great Expectations (проверка на отсутствие дубликатов ключей, допустимые диапазоны значений).

3.    Реализовать паттерн повторных попыток (retry) и оповещения о сбоях в Telegram/Slack.

4.    Заложить основы Data Lineage (зафиксировать зависимости между источниками и сырыми данными).



Часть 3 «Построение Lakehouse и Feature Store».

 Задача: Создать аналитический слой с поддержкой ACID-транзакций и хранилище признаков.

 Действия:

1.    Создать в S3 слой Bronze (сырые данные), Silver (очищенные) и Gold (обогащенные) с использованием Apache Iceberg/Delta Lake через Spark.

2.    Написать трансформации (на SQL или PySpark) для преобразования данных из Silver в Gold-слой (агрегаты, соединения).

3.    Выделить в Gold-слое таблицу признаков (features) для ML-модели (например, признаки студента для прогноза успеваемости).

4.    Развернуть и настроить Feast (Feature Store). Зарегистрировать таблицу признаков, настроить их генерацию и извлечение для обучения.

    Часть 4 «Реализация потокового пайплайна и Real-time дашборда».

    Задача: Организовать обработку событий в реальном времени.

  Действия:

1.    Написать генератор-симулятор событий (например, "студент вошел в аудиторию", "отправил задание"), который пишет в Kafka-топик.

2.    Создать потоковый job на Apache Flink (или с помощью ksqlDB), который агрегирует события в скользящие окна (например, "количество людей в корпусе за последние 5 минут").

3.    Записывать результаты агрегации в колоночную БД (ClickHouse) или сразу в Kafka для чтения.

4.    Настроить реальный дашборд в Grafana, подключив его к ClickHouse/Kafka для отображения метрик.

Часть 5 «Внедрение Semantic Layer и создание Embedded Analytics».

    Задача: Отделить бизнес-логику от данных и встроить аналитику в веб-интерфейс.

    Действия:

1.    Развернуть Cube.js (семантический слой).

2.    Описать в Cube.js data schema: бизнес-логику, измерения и метрики (например, StudentCount, AvgGrade, RoomUtilization).

3.    Создать простой веб-интерфейс на React/Streamlit, который через API Cube.js запрашивает данные и строит кастомные графики.

4.    Реализовать в интерфейсе возможность drill-down (например, с факультета на группу).

   Часть 6 «Полный CI/CD пайплайн платформы и защита проекта».

    Задача: Автоматизировать развертывание и тестирование всей системы.

   Действия:

1.    Создать GitLab CI/CD пайплайн, который:

§  Запускает тесты для кода трансформаций (pytest).

§  Проверяет качество данных (Great Expectations).

§  Собирает Docker-образы для сервисов (Airflow, Cube.js).

§  При успехе деплоит обновления на тестовый стенд (по SSH или в k8s).

2.    Настроить мониторинг пайплайнов данных (логирование, алерт при падении DAG).

3.    Подготовить исчерпывающую документацию: архитектурное решение (ADR - Architecture Decision Record), описание Data Products, инструкции по развертыванию.

4.    Защита: Презентация работающей системы с демонстрацией работы всех компонентов — от симуляции данных до дашборда и алертов.
